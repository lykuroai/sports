## 【完了ステータス】2026-06-26 集約完了

統合サイト化を完了。**公開アプリは `apps/web`（統合サイト）と `apps/admin`（管理画面）の2つのみ**。
旧 `apps/account` / `apps/facility` / `apps/golf` / `apps/outdoor` / `apps/running` は **web へ集約し撤去済み**。
公開サブドメインは **`spotomo.lykuro.ai`（web）/ `admin-spotomo.lykuro.ai`（admin）の2本**。

- **集約ログイン web 内包 済**: account の login/register/phone/auth(callback・line)/billing/
  api/stripe/webhook/users/notifications/settings/verification/withdraw を `apps/web/app` へ移植。
- **施設運営者機能 web 内包 済**: `/owner`・`/owner/{login,register}`・`/facilities/submit`・施設詳細の
  運営者申請を web へ。一般 `/login` と名前空間分離。**ロール累積へは寄せず**、`account_type='facility_owner'`
  の別種別は維持（§1.3 の決定から変更＝下記参照）。
- **golf(GORA 楽天予約)** は固有機能を移植せず**撤去のみ**（DB と facility_sources の出所は残置）。
- **単一オリジン化**: サブドメイン跨ぎの絶対URLログイン誘導は廃止。`NEXT_PUBLIC_ACCOUNT_URL`/
  `FACILITY_URL` は web のオリジン（=`SITE_URL`）、`COOKIE_DOMAIN=.lykuro.ai`（admin が別サブドメインで
  セッション Cookie を共有するため必須）。詳細は CLAUDE.md「統合サイト化」節。
- **インフラ**: `docker-compose.yml` を web/admin/scheduler/caddy の4サービスへ削減、`docker/Caddyfile` を
  2ブロックへ。`NEXT_PUBLIC_*` は build.args に焼き込む（runtime env_file では効かない）。

> DB 集約（§2）は未実施。既存 `facility` スキーマ（`0009`）を継続利用中（§5 の意図的逸脱どおり）。

## 0. 要約

新仕様は「**1サイト・1アカウント・1施設DB・1管理画面**」への集約を求める。現状は
意図的に逆方向（マルチアプリ `apps/{account,golf,running,outdoor,facility,admin,web}` ＋
`*.lykuro.ai` サブドメイン、運営者別アカウント、`facility` スキーマ分割）へ進んでいた。

本移行では現状資産を活かしつつ、**`apps/web` を唯一の統合サイトに育て、種目はパスで出し分ける**。
DB は新仕様の単一 `facilities` 体系へ集約する。**ランニングを最初の縦割り**として完成させ、
golf / outdoor を順次パス配下へ取り込む。

## 1. 目標アーキテクチャ

### 1.1 アプリ構成

| 区分 | 現状(当初) | 実績(2026-06-26 完了) |
|---|---|---|
| ユーザーサイト | `web`(LPのみ) + `golf/running/outdoor`(機能) を別アプリ・別サブドメイン | **`apps/web` 単一**。`spotomo.lykuro.ai`。種目はパス（running=`/running`、他=`/sports/{code}`） |
| アカウント | `apps/account` 集約ログイン | **`apps/web` 内 `/login` 等に内包済み**。`apps/account` 撤去 |
| 施設運営者 | `apps/facility` 別アカウント体系(0024) | **`apps/web` 内 `/owner` に内包済み**。`account_type='facility_owner'` の別種別は**維持**（ロール累積へは寄せず＝§1.3 変更）。`apps/facility` 撤去 |
| 管理画面 | `apps/admin` | **`apps/admin` を併存維持**（`admin-spotomo.lykuro.ai`）。web へは内包せず |
| 共有 | `packages/{api-client,auth-client,domain-common,shared-types,shared-ui,config}` | **そのまま再利用** |

> 旧アプリ（account/facility/golf/outdoor/running）は web へ機能移設後に**撤去済み**。admin のみ別サブドメインで併存。

### 1.2 ルート設計（apps/web）

```text
/                      トップ（種目導線・新着募集・人気カテゴリ・おすすめ施設・開催予定イベント）
/login /signup …       集約ログイン（account の機能を内包）
/running               ランニングカテゴリページ（募集/大会/競技場/コース/地域別）
/golf /outdoor …       各種目カテゴリページ（順次）
/facilities            全施設横断検索
/facilities/[id]       施設詳細
/recruitments          仲間募集 横断検索（種目フィルタ）
/recruitments/[id]     募集詳細
/recruitments/new      募集作成
/events /events/[id]   大会・イベント
/mypage                マイページ
/admin                 管理画面
```

種目別の体験（検索条件・施設属性・募集項目）は `/running` 等のセグメント配下で出し分ける。
現 `apps/running/app/*` は `apps/web/app/running/*` へ移設する。

### 1.3 認証ロールの方針【2026-06-26 実績: ロール累積へは寄せず別種別を維持】

> **当初方針（ロール累積へ戻す）から変更。** 集約時点では `0024` の運営者別アカウント体系を
> **維持したまま web へ内包**した（再設計コストを避け非破壊で移設）。

- 施設運営者は web 内の **`/owner/login`・`/owner/register`** で登録/ログイン（一般 `/login`・`/register`
  と名前空間分離）。種別は `signUp` の `options.data.account_type='facility_owner'`、`handle_new_user`
  トリガーが `account.users.account_type` を設定。
- 運営者専用領域（`/owner`・`/facilities/submit`）は `requireOwnerAccount()` でガード。`apps/web/proxy.ts`
  がパス分岐し owner 系は `loginPath:"/owner/login"`、一般系は `/login`。
- 管理者は `account.user_roles` の `admin` ロールで判定（`requireAdmin()`）。admin は web と別サブドメインの
  ため、未ログインは `${SITE_URL}/login?redirect=<adminの自オリジン>` へ誘導し、共有 Cookie で戻す。
- **未了（将来）**: 仕様§10 のロール累積（guest→user→owner→moderator→admin）への再設計。
  運営者の Google/LINE/電話 登録（現状その経路は `general` になる）。

## 2. DB 集約方針

### 2.1 現状 → 目標

- 現状: `facility` スキーマ（`0009_schema_split.sql`）。カラムは
  `id,name,facility_type,description,postal_code,prefecture,city,address,latitude,longitude,
  geog,source,status,…`。位置は `geog`、出所は `source`(enum)。
- 目標（新仕様）: **単一 `facilities`** ＋ 補助テーブル
  - `facilities`（基本情報・設備フラグ・status・verification_status）
  - `facility_sources`（取得元 `source_type`/`source_id`/`raw_data`/`license`、複数行）
  - `facility_sports`（複数種目 `sport_code` + `is_primary`）
  - `facility_categories`（大分類/小分類）
  - `facility_details` / `facility_sport_details`（種目別はJSONB `detail_data`）
  - `facility_photos` / `facility_change_requests`

### 2.2 移行戦略（非破壊・追加マイグレーション）

1. 新テーブル群を**追加**（既存 `facility.facilities` は残す）。
2. 既存行を新スキーマへ**バックフィル**（`source`→`facility_sources`、`geog`→`latitude/longitude`+`geom`、
   `facility_type`→`facility_categories`/`facility_sports`）。
3. アプリ参照を新テーブルへ切替後、旧スキーマを段階的に縮退。
4. 既存の依存（`nearby_facilities` RPC、`facility_owners` FK、CSV取り込み許可カラム、
   `submitted_data` キー）を新カラム名に追従させる。**CLAUDE.md のスキーマ実態メモを更新**。

> id 型（決定: UUID 維持・依頼者確認済み）: 仕様は `BIGSERIAL` だが、既存 FK・Supabase 認証・
> RLS との整合のため集約テーブルも **UUID（`gen_random_uuid()`）を維持**する。仕様からの
> 意図的逸脱として §5 に記録。

### 2.3 カテゴリ拡張

種目を全スポーツ・レジャー（§3）へ拡張。`sports` ツリー（自己参照）に
golf/running/outdoor 以外（ball_sports/fitness/martial_arts/water_sports/winter_sports/
cycling/leisure/event）を seed 追加。施設の複数カテゴリは `facility_sports` で表現。

## 5. 仕様からの意図的逸脱（記録）

| 項目 | 仕様 | 本実装 | 理由 |
|---|---|---|---|
| 主キー型 | `BIGSERIAL` | UUID 維持 | 既存FK・RLS・Supabase認証との整合 |
| 認証基盤 | 汎用 | Supabase Auth 継続 | 既存資産・RLS前提 |
| 旧アプリ | — | 段階廃止（即時撤去しない） | 移行期の動作担保 |
| 施設カテゴリ/詳細 | `facility_categories`/`facility_details`(平坦) | 既存 `facility_sports`(core.sports ツリー)/`facility_features`(key-value) を継続 | より正規化が進む。Phase2は不足分（取り込み基盤）のみ追加 |

## 6. 留意点（既存メモ継承）

- パッケージ管理は **pnpm**（npm 禁止）。
- 本番デプロイは同一Mac・**main から docker**。Caddy 越しは `X-Forwarded-Host`。
- 通知は必ず `notifyUser()` 経由（サービスロール）。
- 戻り先 redirect は全ログイン手段で一貫保持（`loginUrlFor`/`resolvePostLogin`）。
- **単一オリジン化 済**: web のログインは web 自身で完結。ただし admin は別サブドメインのため
  `COOKIE_DOMAIN=.lykuro.ai` でセッション Cookie を共有、`ACCOUNT_URL=SITE_URL` で admin への
  redirect-back を `resolvePostLogin` の同一 apex 許可で成立させる。
- **`NEXT_PUBLIC_*` はビルド時にバンドルへ inline**されるため、値は `docker-compose.yml` の
  `x-public`(build.args) に置く（`.env.production` の runtime 値だけ変えても効かない）。
- 管理者付与: `account.user_roles` に `admin` ロールを手動 insert（サービスロール）。空だと
  ログインしても `requireAdmin()` が弾きログイン画面へ戻る。
