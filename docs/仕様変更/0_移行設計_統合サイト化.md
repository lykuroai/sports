## 0. 要約

新仕様は「**1サイト・1アカウント・1施設DB・1管理画面**」への集約を求める。現状は
意図的に逆方向（マルチアプリ `apps/{account,golf,running,outdoor,facility,admin,web}` ＋
`*.lykuro.ai` サブドメイン、運営者別アカウント、`facility` スキーマ分割）へ進んでいた。

本移行では現状資産を活かしつつ、**`apps/web` を唯一の統合サイトに育て、種目はパスで出し分ける**。
DB は新仕様の単一 `facilities` 体系へ集約する。**ランニングを最初の縦割り**として完成させ、
golf / outdoor を順次パス配下へ取り込む。

## 1. 目標アーキテクチャ

### 1.1 アプリ構成

| 区分 | 現状 | 目標 |
|---|---|---|
| ユーザーサイト | `web`(LPのみ) + `golf/running/outdoor`(機能) を別アプリ・別サブドメイン | **`apps/web` 単一**。`spotomo.lykuro.ai`。種目はパス |
| アカウント | `apps/account` 集約ログイン | `apps/web` 内 `/login` 等に内包（集約ログインの責務をweb本体へ） |
| 施設運営者 | `apps/facility` 別アカウント体系(0024) | **ロール累積**（guest→user→owner→moderator→admin、仕様§10）へ寄せる |
| 管理画面 | `apps/admin` | `apps/web` 内 `/admin`（または admin アプリを当面併存） |
| 共有 | `packages/{api-client,auth-client,domain-common,shared-types,shared-ui,config}` | **そのまま再利用** |

> 移行期は admin / facility を即時撤去せず、web 側に機能を移し終えてから廃止する（段階廃止）。

### 1.2 ルート設計（apps/web）

```text
/                      トップ（種目導線・新着募集・人気カテゴリ・おすすめ施設・開催予定イベント）
/login /signup …       集約ログイン（account の機能を内包）
/running               ランニングカテゴリページ（募集/大会/競技場/コース/地域別）
/golf /outdoor …       各種目カテゴリページ（順次）
/facilities            全施設横断検索
/facilities/[id]       施設詳細
/recruitments          仲間募集 横断検索（種目フィルタ）
/recruitments/[id]     募集詳細
/recruitments/new      募集作成
/events /events/[id]   大会・イベント
/mypage                マイページ
/admin                 管理画面
```

種目別の体験（検索条件・施設属性・募集項目）は `/running` 等のセグメント配下で出し分ける。
現 `apps/running/app/*` は `apps/web/app/running/*` へ移設する。

### 1.3 認証ロールの方針（決定: ロール累積へ戻す）

仕様§10 の **ロール累積**を採用（依頼者確認済み 2026-06-25）。一般会員が施設運営者も
兼ねられる（guest→user→owner→moderator→admin）。現状 `0024` の運営者別アカウント分離は
**廃止**。`account.users.account_type` は残置（非破壊）しつつ、運営者権限は `user_roles` で
付与する形へ再設計する。`requireOwnerAccount()` は「verified な `facility_owners` 行を持つ
一般会員」を許可する形に変更。`requireGeneralAccount()` ガード（facility 誘導）は撤去。

## 2. DB 集約方針

### 2.1 現状 → 目標

- 現状: `facility` スキーマ（`0009_schema_split.sql`）。カラムは
  `id,name,facility_type,description,postal_code,prefecture,city,address,latitude,longitude,
  geog,source,status,…`。位置は `geog`、出所は `source`(enum)。
- 目標（新仕様）: **単一 `facilities`** ＋ 補助テーブル
  - `facilities`（基本情報・設備フラグ・status・verification_status）
  - `facility_sources`（取得元 `source_type`/`source_id`/`raw_data`/`license`、複数行）
  - `facility_sports`（複数種目 `sport_code` + `is_primary`）
  - `facility_categories`（大分類/小分類）
  - `facility_details` / `facility_sport_details`（種目別はJSONB `detail_data`）
  - `facility_photos` / `facility_change_requests`

### 2.2 移行戦略（非破壊・追加マイグレーション）

1. 新テーブル群を**追加**（既存 `facility.facilities` は残す）。
2. 既存行を新スキーマへ**バックフィル**（`source`→`facility_sources`、`geog`→`latitude/longitude`+`geom`、
   `facility_type`→`facility_categories`/`facility_sports`）。
3. アプリ参照を新テーブルへ切替後、旧スキーマを段階的に縮退。
4. 既存の依存（`nearby_facilities` RPC、`facility_owners` FK、CSV取り込み許可カラム、
   `submitted_data` キー）を新カラム名に追従させる。**CLAUDE.md のスキーマ実態メモを更新**。

> id 型（決定: UUID 維持・依頼者確認済み）: 仕様は `BIGSERIAL` だが、既存 FK・Supabase 認証・
> RLS との整合のため集約テーブルも **UUID（`gen_random_uuid()`）を維持**する。仕様からの
> 意図的逸脱として §5 に記録。

### 2.3 カテゴリ拡張

種目を全スポーツ・レジャー（§3）へ拡張。`sports` ツリー（自己参照）に
golf/running/outdoor 以外（ball_sports/fitness/martial_arts/water_sports/winter_sports/
cycling/leisure/event）を seed 追加。施設の複数カテゴリは `facility_sports` で表現。

## 3. 外部データ取得バッチ（Phase 2 以降）

`facility_data §11` の通り、Google Places / OSM Overpass / 自治体オープンデータ /
国土数値情報 / 楽天GORA を**定期バッチ**で取得→正規化→重複判定→承認。MVP では
ランニング関連（陸上競技場・ランニングコース・運動公園）を OSM/自治体データ中心に先行。
既存 `apps/running/app/api/cron/sync-races`（races バッチ）を `batch_runs` 体系に寄せる。

## 4. フェーズ計画（ランニング先行）

### Phase 1: 統合サイト土台＋ランニング移設（最優先）
1. `apps/web` に集約ログイン/proxy/共有レイアウト（ヘッダー/フッター=ブランド線画）を実装。
2. トップページを仕様§4.2の導線（種目から/地域から/現在地/施設から/募集から）へ刷新。
3. `apps/running/app/*` を `apps/web/app/running/*` へ移設し動作担保。
4. `/facilities` `/recruitments` `/events` `/mypage` を web 直下に統合（種目フィルタ付き）。

### Phase 2: 施設DB集約
5. 統合 `facilities` スキーマのマイグレーション＋バックフィル。
6. 施設検索/詳細/提案/CSV取り込み/管理画面を新スキーマへ切替。

### Phase 3: 外部データバッチ ✅（OSM 済 / 自治体は今後）
7. OSM・自治体オープンデータ取得バッチ（ランニング/公園系）→正規化→重複判定→承認。
   - 済: OSM(Overpass) 取り込み `apps/web/lib/osm-sync.ts` ＋ `/api/cron/sync-osm-facilities`。
     `find_duplicate_candidates` で重複判定、未承認(`status='unverified'`)登録、`facility_sources`
     に ODbL 帰属＋raw 記録、`core.batch_runs` に実行履歴。施設一覧は verified のみ表示。
   - 済: OSM を bbox クエリ化（ミラー互換）＋複数県へ拡大（PREFECTURE_BBOX/OSM_FETCH_AREAS）。
     scheduler(compose) で週次自動実行。本番デプロイ済・初回取り込み実行済。
   - 済: 承認UI（apps/admin `/facilities`：未承認一覧＋出所＋重複候補＋承認/却下＋**統合(merge)**＋
     **県フィルタ/ページング/一括承認**）と `/batch-runs`（履歴）。
   - 済: 自治体オープンデータ(CKAN/CSV)取り込み（`/api/cron/sync-municipal-facilities`。
     文字コード自動判定・列柔軟推定・BODIK で検証）。
   - 今後: 国土数値情報・Google Places、施設詳細での OSM/出所 帰属表示、自治体ソース一覧の
     scheduler 連携（複数URL定期実行）。

### Phase 4: 種目横展開
8. golf（楽天GORA連携を `/golf` 配下へ）、outdoor を web 配下へ取り込み。
9. ball_sports / fitness / leisure 等のカテゴリページとマスター追加。

### Phase 5: 認証ロール統合・旧アプリ撤去
10. 運営者をロール累積へ再設計（要確認後）。account/facility/admin/golf/outdoor/running アプリを廃止。

## 5. 仕様からの意図的逸脱（記録）

| 項目 | 仕様 | 本実装 | 理由 |
|---|---|---|---|
| 主キー型 | `BIGSERIAL` | UUID 維持 | 既存FK・RLS・Supabase認証との整合 |
| 認証基盤 | 汎用 | Supabase Auth 継続 | 既存資産・RLS前提 |
| 旧アプリ | — | 段階廃止（即時撤去しない） | 移行期の動作担保 |
| 施設カテゴリ/詳細 | `facility_categories`/`facility_details`(平坦) | 既存 `facility_sports`(core.sports ツリー)/`facility_features`(key-value) を継続 | より正規化が進む。Phase2は不足分（取り込み基盤）のみ追加 |

## 6. 留意点（既存メモ継承）

- パッケージ管理は **pnpm**（npm 禁止）。
- 本番デプロイは同一Mac・**main から docker**。Caddy 越しは `X-Forwarded-Host`。
- 通知は必ず `notifyUser()` 経由（サービスロール）。
- 戻り先 redirect は全ログイン手段で一貫保持（`loginUrlFor`/`resolvePostLogin`）。
- 統合に伴いサブドメイン横断の `loginUrlFor` は不要化する方向（単一オリジン化）。
